# CLAUDE.md — gourmet-sp3 開発ガイド

## プロジェクト概要

グルメAIコンシェルジュアプリ。Gemini LiveAPIによるリアルタイム音声対話 + A2Eリップシンクアバター。
- **フロントエンド**: Astro + TypeScript
- **バックエンド**: Flask + Socket.IO (Python) → Cloud Run
- **AI**: Gemini LiveAPI (音声対話), Gemini REST API (テキストチャット・ショップ検索)
- **アバター**: LAM/A2E (Audio2Expression) — 52個のblendshape係数による低遅延リップシンク

---

## ⚠️ 知識ベースの限界宣言 — 最重要

**本プロジェクトの中核技術は、全てClaudeの知識ベース外にある。**

| 技術 | リリース時期 | Claudeの知識 |
|------|------------|-------------|
| LAM/A2E (Audio2Expression) | 2025年10月（アリババ研究所論文） | **なし** |
| Gemini LiveAPI | 2025年12月末（プレビュー版） | **なし** |
| `gemini-2.5-flash-native-audio-preview` | 2025年12月末 | **なし** |
| LiveAPIのfunction calling仕様 | 2025年12月末 | **なし** |

### これが意味すること

1. **Claudeが「一般的にはこうだ」と思う知識は、このプロジェクトでは通用しない**
2. **気を利かせた修正 = 古い常識の押し付け = プロジェクトの破壊**
3. **推論で進めると必ず間違う。間違いを推論で修正するとさらに間違う。これが泥沼化の構造**
4. **唯一の正解は「コードを読む」「仕様書を読む」「ユーザーに聞く」の3つだけ**

### 実際に起きた泥沼化の構造（20日間・300時間超の損失）

```
Claudeの古い知識で「改善」
  → 新技術の設計思想に反する修正
    → 動かなくなる
      → 原因を古い知識で推論
        → 的外れな追加修正
          → さらに壊れる
            → 3歩進んで2歩下がる（場合によっては1歩進んで3歩下がる）
```

**この連鎖を断ち切るのがこのCLAUDE.mdの目的。**

## 絶対守るべきルール

### 1. 推論するな。確認しろ。

- **知識ベースにない技術（Gemini LiveAPI, LAM/A2E等）について推測で語るな**
- コード1行読めばわかることを推論で済ませるな
- 「可能性としては…」「おそらく…」は禁止。事実だけ述べろ
- わからないことは「わかりません」と言え

### 2. 仕様書を先に読め

- `docs/` 配下の仕様書を**必ず先に読んでから**作業を開始する
- 設計意図が明記された判断を「問題」と指摘するな
- ユーザーの実証テスト結果はパラメータ値や設計判断の根拠。安易に否定するな

### 3. 勝手に修正するな

- **コード修正は必ずユーザーの許可を得てから実行する**
- 特に以下は絶対に確認なしで変更するな:
  - モデル名・モデルバージョン
  - 本番設定（環境変数、デプロイ設定）
  - CI/CDワークフロー
  - GCS上のプロンプトファイル
  - セキュリティ・IAM関連

### 4. フォールバック禁止 — 本プロジェクト最大の地雷

- 問題の根本原因を特定・修正せよ
- フォールバック機構で問題を覆い隠すな
- キーワード検出による代替ロジックは絶対にやるな

**なぜフォールバックが特に危険か（本プロジェクト固有の理由）:**

```
本プロジェクトは実証テストフェーズ。
フォールバック = 「新技術が動かないなら古い方式で代替」
           = Claudeの知識ベースにある古いロジックへの退行
           = 世界最先端の新技術を検証するプロジェクトの目的そのものの否定
```

- 本番運用にはフォールバックは必要だが、**今はその段階ではない**
- 「安全に動かすため」のフォールバックが、プロジェクトの存在意義を台無しにする
- フォールバックを入れた瞬間、新技術の問題が隠蔽され、根本修正の機会を失う

### 5. 指示・回答は最後まで読め

- ユーザーの指示を部分的に読んで着手するな
- Gemini等LLMの回答も全て読んでから実装せよ
- 5つの対策が提示されたら5つ全て理解してから動け

### 6. 根本原因を追え

- 副次的症状に飛びつくな（例: 発火しない問題を放置して1008エラーに対処するな）
- 修正前に正常だったなら、修正が原因。推論で別の原因を探すな
- ユーザーの「修正前は正常だった」という事実報告は最優先の手がかり

## アーキテクチャの要点

### プロンプト管理（GCS一元管理）

- **正式なプロンプトはGCS上のファイルに一元管理**（`prompts/concierge_ja.txt` 等）
- 実証テスト中の仮説検証用パッチのみ、Pythonベタ書きを暫定的に認める
  - `live_api_handler.py` の `LIVEAPI_*` 定数（LiveAPI用）
  - `support_core.py` の `json_enforcement`（REST API JSON強制）
- **暫定パッチはGCSプロンプトと矛盾してはならない**
- 安定確認後はGCS版へ統合すること
- GCSファイルの修正にはIAMロール・セキュリティの制約がある。勝手にデプロイワークフローを変えるな

### ショップ検索の発火方式

- コード側でAI発話テキストからキーワード検知する処理は**完全に廃止済み**
- `should_trigger_shop_search()`, `SHOP_TRIGGER_KEYWORDS` は削除済み
- **conciergeモード**: 検索判断は**LLMに一任**。条件が揃ったとLLMが判断→FC発火
- **chatモード**: 1ターン目は条件1つで**即検索**。2ターン目以降はLLMが判断（深掘り or 再検索）
- JSON出力はショップカード表示時のみ。通常の会話・深掘りは自然な文章

### A2E (Audio2Expression) リップシンク

- **フレームは送らない**。52個のblendshape係数だけを送る — これがA2Eの核心
- バッファ閾値（初回0.1秒、後続5秒）は実証テスト済みの値。変更するな
- 詳細は `docs/09_liveapi_migration_design_v6.md` §4 を参照

### LiveAPIセッション管理

- セッションタイムアウト時は再接続 → 会話履歴再送（`_send_history_on_reconnect()`）
- `tool_config mode="ANY"` はLive APIでは**利用不可**（公式ドキュメント確認済み）
- `send_tool_response()` はキーワード引数 `function_responses=[...]` で渡す（SDK仕様）

## コード修正前の必須チェックリスト

**修正に着手する前に、以下を必ず順番に実行せよ。省略禁止。**

### Step 1: 仕様書を読んだか？
- 該当機能の仕様書（`docs/`配下）を読んだか
- 設計意図・パラメータ値の根拠が書かれていないか確認したか
- → 読んでいないなら、修正を始めるな

### Step 2: 現在のコードを読んだか？
- 修正対象のファイルを実際に読んだか（推論で「こうなっているはず」は禁止）
- 修正前の状態を正確に把握したか
- → 読んでいないなら、修正を始めるな

### Step 3: 自分の知識ベースにある技術か？
- Gemini LiveAPI、LAM/A2E、最新SDK仕様に関わる修正ではないか
- → **知識ベース外の技術なら、推論での修正は禁止。ユーザーに確認せよ**

### Step 4: ユーザーの許可を得たか？
- 修正内容をユーザーに提示し、許可を得たか
- → 特にモデル名、プロンプト、設定値、CI/CDは確認必須

### Step 5: 修正は1つだけか？
- 複数の修正を同時に入れていないか
- → 1つずつ修正し、各修正の効果を確認できるようにせよ

## 禁止パターン集 — 過去セッションで実際に発生した失敗

### ❌ パターン1: 「一般的にはこうだから」修正
```
状況: A2Eのバッファ閾値が初回0.1秒・後続5秒
Claude: 「0.1秒は極端に小さい、5秒は極端に大きい」と批判
事実: 両方とも実証テスト済み。公式ドキュメントにも明記
教訓: 自分の「一般的な」感覚でパラメータを批判するな
```

### ❌ パターン2: 知らない技術を知っているふりで分析
```
状況: A2Eリップシンクのレイテンシ分析を依頼された
Claude: 「構造的レイテンシ2-4秒」と診断
事実: 修正前は遅延ゼロ。アリババの公式デモでも遅延なし
教訓: 知識ベースにない技術について「構造的に○○」と断言するな
```

### ❌ パターン3: フォールバックの自動挿入
```
状況: LiveAPIのfunction callingが発火しない
Claude: 「AIの発言から『お調べ』『お探し』を検出して検索発動」を実装
ユーザー: 「昭和のAIか？」
教訓: LLM判断方式を捨ててキーワードマッチングに退行するな
```

### ❌ パターン4: 確認なしの本番設定変更
```
状況: Geminiの回答に「モデル変更」の選択肢があった
Claude: 確認なしにモデルを変更してコミット&プッシュ
教訓: モデル名、環境変数、CI/CDは絶対に確認してから
```

### ❌ パターン5: 的外れ修正の連鎖（モグラたたき）
```
状況: 1つの問題（function calling不発火）に対して
Claude: 9回の的外れな修正を連鎖 → 最終的にユーザーが根本原因を指摘
教訓: 2回修正して直らなかったら立ち止まれ。根本原因を再考せよ
```

### ❌ パターン6: プロンプト配信経路の混同
```
状況: プロンプトを修正したつもり
Claude: Pythonハードコードだけ修正し、GCSのプロンプトファイルは未修正
結果: 本番では古いプロンプトが使われ続けた
教訓: GCS（REST API用）とPythonハードコード（LiveAPI用）は別系統
```

### ❌ パターン7: LLMの回答を部分的にしか読まない
```
状況: Geminiが5つの対策を提示
Claude: 1つ目だけ実装、2つ目はパラメータを取り違えて実装
教訓: 回答は全部読んでから着手。部分的な着手は取りこぼしの元
```

## 作業時の心得

1. **確認 → 事実 → 行動** の順序を守る
2. **公式ドキュメントを最初に確認**する（LLMの回答ではなく）
3. **LLMにLLMの動作保証を求めても意味がない**
4. 1つの問題に対して的外れな修正を連鎖させるな。立ち止まって根本原因を考えろ
5. 「ちょっと確認すれば済むこと」を推論で済ませるな — それが泥沼の入口
6. **2回修正して直らなかったら、手を止めてユーザーに相談せよ**
7. **「気を利かせる」な。指示された通りにやれ。**

## Claudeの役割制限 — 「道具」として使え

### Claudeにやらせて良いこと
- ユーザーが特定した修正を、指示された通りに実行する
- コードを読んで、事実だけを報告する（「この関数は○行目にある」「この変数の値は○○」）
- git操作（commit, push, tag等）

### Claudeにやらせてはいけないこと
- **問題の分析・診断**（間違うから）
- **改善提案**（知識ベース外の技術だから）
- **CLAUDE.md、docs/、仕様書の編集**（ガードレールの改ざんになるから）
- **複数ファイルにまたがる修正**（影響範囲を正確に把握できないから）
- **「ついでに」の修正**（指示されていないことはやるな）

### 運用ルール
1. ユーザーが原因を特定する
2. ユーザーが修正内容を指示する（ファイル名・行番号・変更内容）
3. Claudeは指示された通りに修正を実行する
4. 1修正1コミット。修正後はユーザーが動作確認する
5. **2回修正して直らなかったら、Claudeは手を止めてユーザーに報告する**
