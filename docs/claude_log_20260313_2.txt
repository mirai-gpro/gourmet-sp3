提案への影響
2段階分割する場合、ソフトリロード時の扱いが変わります：

リセットボタン（ユーザージェスチャーあり）：

AudioContextが生き残っていれば → resume() して即LiveAPI再接続 + 挨拶再生 → OK

バックグラウンド復帰（ユーザージェスチャーなし）：

AudioContextが生き残っていれば → resume() はジェスチャーなしでも成功する場合がある（iOS Safari は既にresumed済みのAudioContextの再resumeは許可する傾向）

結論：ソフトリロード時はAudioContextを破棄しないのが正解
ソフトリロード時：
  1. LiveAPIセッションだけ切断（live_stop）
  2. AudioContext/MediaStreamは保持（terminate()しない）← 今と同じ
  3. 新しいLiveAPIセッション開始（live_start）
  4. greeting trigger → 既存のAudioContextで再生 → OK
  5. マイクも既存のMediaStreamが生きている → startStreaming()するだけ

初回: initPlayback() → スプラッシュtap → resume() → 挨拶再生 / マイクclick → initMicrophone()
ソフトリロード: AudioContext生存 → terminate() せずLiveAPIだけ再接続 → 挨拶再生 + startStreaming() で即マイク復帰
initialize() の if (this.audioContext) return; ガードが、ソフトリロード時にAudioContext再利用を保証してくれるので、この設計と相性がいいです。

質問への回答
1 & 2. テキストと音声入力の混在とタイミングについて

混在は可能か？: はい、可能です。send_client_content（テキスト）と send_realtime_input（音声）は同一セッション内で自由に混在できます。
タイミングの制約: ダミーテキストに対するAIの応答（音声）が完了するのを待つ必要はありません。AIが喋っている最中に音声を送り始めても、API側のVADが「ユーザーの発話開始」を検知した時点で、AI側の音声生成は中断（Interrupted）され、新しいターンに移行します。これがこのAPIの「全二重（Full-duplex）」の強みです。
ストリーミング停止（ポーズ）の影響
状態の問題: クライアント側で音声送信を止めても、WebSocketが維持されていればLive API側のコンテキストは維持されます。
タイムアウト: 無音状態が続くと約5〜10分でセッションが切断されますが、数秒〜数十秒のボタン操作待ち程度では問題ありません。VADの状態もリセットされず、送信再開後の最初の有効な音声から検知が始まります。
発話開始のトリガー
turn_complete の要否: send_realtime_input（音声入力）の場合、turn_complete を送る必要はありません。API側のVADが「静寂（silence_duration_ms）」を検知すると、自動的にそのターンを締め切り、応答を開始します。
※逆にテキスト入力（send_client_content）の場合は、明示的に turn_complete: True が必要です。
音声フォーマットの詳細と不一致
期待されるフォーマット: 16kHz / 16-bit Signed PCM / Little Endian / Mono です。
不一致の影響: サンプルレートが異なると、AIには「早口」または「スロー」な音声として聞こえ、認識率が著しく低下します。ビット深度が異なると、単なるノイズとして処理され、AIが一切反応しない「サイレントエラー」になります。
iOS Safari の getUserMedia 失敗
サイレントエラーの可能性: 非常に高いです。iOSでは「ユーザー操作（clickイベント等）の直系コールスタック」以外でのマイク起動や AudioContext.resume() は、Promiseがリジェクトされるか、静かに無視されます。ページロード時の自動実行は確実に失敗します。
リップシンク（音声とテキストの同期）
同期方法: server_content メッセージ内の model_turn に含まれる inline_data（音声バイナリ）と、output_audio_transcription（テキスト）はほぼ同時にストリームされます。
精度: 音声チャンクのバイト数から再生時間を逆算（1秒=48,000バイト ※24kHz/16bit想定）し、届いたテキストを表示するタイミングをキューイングすることで、実用的なリップシンクが可能です。
バグ修正案
問題1: マイクボタンが1回目で機能しない原因と修正
原因: initializeSession() で自動的に LiveAPI モードを開始しようとしているため、iOS/ブラウザ側で AudioContext が suspended 状態（ロック状態）のまま固定されています。また、1回目のクリックで terminateLiveSession が走っていた名残で状態が不安定になっています。
修正案:
「自動開始」をやめる: initializeSession では REST API のセッションID取得のみ行います。
初回のマイクボタンクリックをトリガーにする:

// toggleRecording 内 if (!this.isLiveMode) { // 1回目のクリック時：ここで初めてブラウザの権限とコンテキストを有効化 await this.enableAudioPlayback(); // AudioContext.resume() を含む await this.startLiveMode(); // ここで getUserMedia と Socket.IO 接続 return; // startLiveMode 内で挨拶が始まるので、ここでは録音状態にしない }
