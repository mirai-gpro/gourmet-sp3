処理ステップデータ形式サンプリングレート役割1. LiveAPI出力LPCM (Base64)24,000 HzGeminiの生の声2. 分岐（直列）LPCM (Binary)24,000 Hzブラウザ再生とA2E送信用に分離3. リサンプリングLPCM (Binary)16,000 HzA2E(LAM)の要求仕様に適合4. A2E入力LPCM (Tensor/Array)16,000 Hz表情パラメータの生成
Google スプレッドシートにエクスポート
なぜMP3ではなく「LPCMリサンプリング」なのか？
A2Eのモデル特性: A2E（LAM）の内部で使われている音声特徴抽出器（HuBERT等）は、16kHzのLPCMを前提に学習されています。MP3の圧縮ノイズは、口の動きの滑らかさを損なう原因になります。
遅延（レイテンシ）: MP3エンコードには「フレームの蓄積」が必要なため、数十ミリ秒の遅延が必ず発生します。LPCMのリサンプリングであれば、1サンプル単位で処理できるため、遅延をほぼゼロに抑えられます。
実装の単純化: LiveAPIから届くデータはすでにPCMです。これをMP3にパックして、またA2E側でデコードしてPCMに戻すのは、計算リソースの無駄です。
ClaudeCode への具体的な指示（プロンプト例）
ClaudeCodeに対し、以下の設計方針で実装するよう指示してください。
ClaudeCodeへの指示案: 「Gemini LiveAPI (24kHz LPCM) から LAM_Audio2Expression (16kHz LPCM) へのパイプラインを構築したい。
MP3への変換は行わないこと。
LiveAPIから届く24kHzのBase64 LPCMをデコードし、16kHzへダウンサンプリングする処理を実装せよ。
Python（バックエンド）で処理する場合、librosa.resample または scipy.signal.resample_poly を使用して高速にリサンプリングせよ。
リサンプリング後の16kHz PCM（16bit signed）をそのまま A2E の入力関数に渡す設計にせよ。」
バックエンド（Python）でのリサンプリング実装例
もしサーバー側（Flask/Cloud Run）でA2Eを動かしているなら、以下の処理が最も高速です。
Python
import numpy as np from scipy import signal def prepare_audio_for_a2e(pcm_24k_data): # pcm_24k_data: int16のバイト列 # 1. バイト列をnumpy配列(float32)に変換 audio_fp32 = np.frombuffer(pcm_24k_data, dtype=np.int16).astype(np.float32) / 32768.0 # 2. 24kHzから16kHzへリサンプリング (24:16 = 3:2) # resample_polyは高速でストリーミング向き audio_16k = signal.resample_poly(audio_fp32, 2, 3) return audio_16k # これをそのままA2Eモデルへ
