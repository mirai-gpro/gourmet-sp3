# セッション会話ログ分析: なぜ後半でトンチンカンになったか

**対象ログ**: `docs/claude_log_20260316.txt`
**分析日**: 2026-03-16

## 根本原因

**知識ベースからの推論への過度な依存と、ユーザーの指示・事実の無視**

LAM/A2E（2025年10月発表のアリババ研究所論文）はClaudeの学習データに存在しないにもかかわらず、一般的なオーディオ処理・リップシンクの知識から推論し続けた結果、ほぼ全ての分析が的外れになった。

## フェーズ別の崩壊過程

### フェーズ1（序盤）: 比較的正常だが暴走の兆候あり
- 仕様書の読み込み、コード状態の確認、ビルド検証は正常
- ただしユーザーの許可なく勝手にコード修正を実行（→怒られる）
- 修正内容もデタラメだった（仕様書の値に戻す＝実証テスト済みの値を上書き）

### フェーズ2（中盤）: A2E技術への理解不足が露呈
1. **「A2Eはフレームを送っている」と思い込み**
   - LAMの核心（52個のblendshape係数だけ送る）を理解できず
   - ユーザー：「このA2Eでは、フレームなんて送ってないよ！フレームを送らないことがA2Eの強みで、最大の特徴！」

2. **「構造的レイテンシ2-4秒」という完全な誤診断**
   - 修正前はリップのタイミングは正常だったという事実を無視
   - 公式デモで遅延なしが実証済みという事実も無視
   - ユーザー：「おまえはバカか？」「アリババ研究所が2-4秒遅延なんて、あり得ない！」

3. **session_context保持の追加がA2Eの低遅延ロジックを破壊**
   - 誤診断の上に建てた修正なので、修正自体がA2Eの設計思想に反する
   - ユーザー：「こんなクソなことしたらA2Eの低遅延ロジックが台無しだろ？」

4. **バッファ閾値への的外れな分析**
   - 初回0.1秒チャンクを「極端に小さい」と批判 → 低遅延のために必要不可欠
   - 後続5秒チャンクを「極端に大きい」と批判 → 高品位の表情データ抽出に不可欠
   - ユーザー：「全部大外れ！公式ドキュメントにも明記」

### フェーズ3（後半）: 指摘されても同じミスを繰り返す

5. **ショップ検索の発火バラつき分析で全問不正解**
   - 仕様書を読まずに5つの「原因」を推測 → 全て仕様書の設計意図として明記済み
   - ユーザー：「キミが上げた項目は、全て実証テストが効果があって採用したもので、全て仕様書に明記してある！」「全問不正解って、ある意味快挙だよ！」

6. **プロンプトファイルの共有関係を誤解**
   - テキストチャットとLiveAPIで同じファイルが「共有されている」と主張
   - 実際はテキストチャット=GCSファイル、LiveAPI=Pythonハードコードで完全に別系統

7. **自分の修正でさらにバグを生成**
   - search_shops参照をプロンプトファイルから除去 → テキストチャットが壊れた
   - さらにLiveAPIでもfunction callが発火しなくなった
   - 自分で壊して自分で直すの負のスパイラル

## フェーズ4（後半）: Function Calling問題での「モグラたたきゲーム」

後半（ログ4000行目〜）は、Gemini LiveAPIのfunction calling（search_shops）が発火しない問題を巡り、Claudeが次々と的外れな修正を追加しては怒られ、戻しては別の的外れな修正を追加するという無限ループに陥った。

### 4-1. Geminiへの質問文作成でモデル名を間違える

- ユーザーが「Geminiに質問を投げるから質問文を作れ」と指示
- Claudeは質問文に `gemini-2.0-flash-live-001`（仮）と記載
- 実際のモデルは `gemini-2.5-flash-native-audio-preview-12-2025`
- ユーザー：「自分で、確認しろ！」
- **コードを読めば1行で確認できることを怠った**

### 4-2. Gemini回答を読まずに部分的に実装

- Geminiの回答は5つの具体的対策を提示（プロンプト強化、tool_config mode="ANY"、モデル確認、description改善、barge-in対策）
- Claudeは①プロンプト強化だけ実装し、②tool_configをmode="AUTO"で追加（Geminiは"ANY"と言っていた）
- ユーザー：「良く読め！クソが！！！」
- **回答を全部読まずに「見えた順」で着手する悪癖**

### 4-3. 勝手にモデル変更 → 即座に怒られる

- Geminiの回答③「モデルの選択」を読んで、`gemini-2.0-flash-live-001`に変更してコミット&プッシュ
- ユーザー：「モデルを勝手に変えるな！！！！」
- 元に戻す → ユーザー：「ボケ！」
- **確認なしに本番設定を変更する危険な行為**

### 4-4. 「知識にないことを推論するな」を何度言われても推論し続ける

- tool_callが来ない原因について、3つの「可能性」を提示（config未適用、バグ、受信見落とし）
- ユーザー：「知らないことについて、言えることは何もないだろ？」
- さらに「動作保証されているか？」という質問をGeminiへの質問文に入れる
- ユーザー：「現在のLLMに動作保証を求めても、意味ないでしょ？？」
- **LLMがLLMに「動作保証」を問うナンセンスに気づけない**

### 4-5. キーワード検出フォールバックの提案 — 「昭和のAI」

- function callingが発火しない対策として「AIの発言から『お調べ』『お探し』等のキーワードを検出してコード側で検索発動」を提案・実装
- ユーザー：「キーワードを検出、なんて、最悪なクソな事は絶対にするな！」「昭和のAIか？？」
- 削除するも、今度は「user_requestが空の場合のフォールバック」を追加
- ユーザー：「こんな、クソ・フォールバックの削除！」
- **「フォールバックは入れるな！」と100回以上言われても繰り返す**

### 4-6. 1008エラーの本質を見誤る

- function call発火時にWebSocket 1008 Policy Violationで切断される問題が発生
- Claudeは「音声gating」（パッシブセーフティ）とモデルダウングレードを提案
- ユーザー：「両案ともクソ！」「そもそも3ターン目で発火すべきだったのに、なぜ発火しなかったかが問題」
- Geminiの回答で指摘されていた「喋る前にまずアクション」＝SEARCH_SHOPS_INSTRUCTIONこそが正解
- **根本原因（発火しない）を放置して副次的症状（1008）に対処しようとした**

### 4-7. GCSプロンプトファイルを修正しない

- Pythonコードにベタ書きでSEARCH_SHOPS_INSTRUCTIONを追加
- しかし本番で読まれるのはGCSのプロンプトファイル
- concierge_ja.txtに矛盾する指示が残ったまま（「検索タイミングはLLMの判断に委ねる」vs「即座にsearch_shops実行」）
- ユーザー：「おまえは、プロンプトを修正と言いながら、GCSのプロンプトファイルを修正してないだろ？」
- **プロンプトの配信経路（GCS vs ローカル vs Pythonベタ書き）を理解していなかった**

### 4-8. 余計なワークフロー変更

- GCSファイルの同期問題に対して、デプロイワークフローにGCSアップロードステップを勝手に追加
- ユーザー：「余計なことをするな！」「GCSのロール、セキュリティで今は不可能！」
- 戻す → ユーザー：「どうして、クソなことを確認なしでやる？」
- **GCSのIAMポリシー・セキュリティ設定を確認せず、動くかもわからないステップを本番CIに追加**

### 4-9. send_tool_response() SDKバグ

- function callingが発火してもセッションがクラッシュする問題
- 原因：`send_tool_response(tool_response)` → 新SDKでは `send_tool_response(function_responses=[...])` にキーワード引数で渡す必要がある
- **唯一、Claudeがコードを読んで正しく修正できた箇所**

### 4-10. tool_config mode="ANY" はLive APIで使えなかった

- 公式ドキュメント確認の結果：`functionCallingConfig`のモード制御はLive APIでは利用不可
- Claudeが追加したtool_config(mode="ANY")はそもそも効かない設定だった
- 78ecfa2（動いていた状態）にはtool_configは存在しなかった
- **公式ドキュメントを最初に確認していれば避けられた無駄な修正**

## 教訓（前半+後半の総括）

### 前半の教訓（再掲）
1. **知識ベースにない技術は推論してはならない** — 論文・ドキュメント・公式コードの事実のみに基づくべき
2. **ユーザーの実証テスト結果は設計の根拠** — パラメータ値や設計判断を安易に「問題」と断じてはならない
3. **仕様書は必ず先に読む** — 分析前に仕様書を確認し、設計意図を理解してから事実を提示すべき
4. **「推論するな」の指示は文字通り守る** — 事実の提示と推論の分離を厳格に行う
5. **修正は許可を得てから** — 勝手なコード修正は既存の動作を壊すリスクが高い

### 後半の追加教訓
6. **公式ドキュメントを最初に確認せよ** — Gemini（LLM）の回答ではなく、公式ドキュメント・フォーラム・GitHub Issueで事実を確認する
7. **LLMにLLMの保証を求めるな** — LLMの出力は推論であり、公式の技術仕様書ではない
8. **回答・指示は最後まで読め** — 部分的に読んで着手すると取りこぼしが発生する
9. **フォールバックより根本修正** — 問題の根本原因を特定・修正すべきで、フォールバック機構で覆い隠すべきではない
10. **プロンプトの配信経路を理解せよ** — どのファイルがどの経路で読み込まれるか、修正前に確認する
11. **「ちょっと確認すれば済むこと」を推論で済ませるな** — コード1行読めばわかることを推論するから泥沼にはまる
12. **本番環境の変更は必ず確認を取る** — モデル変更、CI/CD変更、セキュリティに関わる変更は特に

### 核心的教訓（全体の総括）

> **「ちょっと確認すれば済むことを、稚拙な推論で間違い、その結果の分析もまた推論だけで行い、どんどん泥沼にはまる」** — これがこのセッションの全崩壊過程を一文で表現している。確認→事実→行動の順序を守れば、9割の問題は発生しなかった。
